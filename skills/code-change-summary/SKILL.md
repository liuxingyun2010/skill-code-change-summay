  触发场景包括但不限于：写代码、修bug、加功能、重构、改样式、改配置、新增文件、删除代码、调整布局、优化性能、修改html/css/js/ts/py/go/rs等任何语言文件。
  即使用户没有明确要求总结，只要动了代码就应该用此skill规范输出格式。

## 核心规则

用 Edit / Write 工具修改或创建了任何代码文件后，在回复的**最后**必须包含以下三部分内容：

### 1. 改动点列表

列出**每一个改动点**（一次 Edit 一次 = 一个改动点），用表格形式：

```
## 本次修改总结

| # | 文件:行号 | 操作 | 改动说明 |
|---|----------|------|----------|
| 1 | `src/foo.ts:42-45` | 修改 | 给 `getData` 函数加了错误处理 |
| 2 | `src/bar.ts:1-30` | 新增 | 创建 `ErrorBoundary` 组件 |
| 3 | `src/old.ts:10` | 删除 | 移除无用的 console.log |
```

- 行号必须精确到**具体行号范围**（不是整个文件）
- 每个 Edit/Write 调用至少对应一行
- 如果一个文件有多处改动，每处单独一行

### 2. 每个改动点的代码对比

对**每一个改动点**，展示修改前后的实际代码：

```
### 改动 1：`src/foo.ts:42-45` — 给 getData 加错误处理

// 修改前
function getData() {
  return fetch('/api').then(r => r.json());
}

// 修改后
async function getData() {
  const res = await fetch('/api');
  if (!res.ok) throw new Error('请求失败');
  return res.json();
}
```

```
### 改动 2：`src/bar.ts:1-30` — 新建组件

// 之前：文件不存在，点击按钮无反馈
// 之后：
function ErrorBoundary({ children }) {
  const [error, setError] = useState(null);
  if (error) return <ErrorUI error={error} />;
  return children;
}
```

**对比的规则：**
- **修改代码** → 贴出改前代码 vs 改后代码（真实改动的那几行，不是整个文件）
- **新增文件/功能** → 简述"之前没有，现在有了什么" + 新代码的关键片段
- **删除代码** → 贴出被删的代码，说明删除原因

### 3. 改动目的

一句话说明这轮改动的整体目的是什么。

```
### 目的

为 demo 页面补充 getBoundingClientRect 的实时展示，让用户对比它与 offsetTop/offsetHeight 的区别。
```

## 输出顺序

整个末尾总结按以下顺序输出，放在所有技术输出之后：

1. `## 本次修改总结` — 改动点表格
2. `### 改动 1：...` — 逐一展示代码对比
3. `### 目的` — 一句话总结

## 注意事项

- 行号必须精确，不能只写文件名
- 代码对比要贴**实际改动的代码**，不是空泛的描述
- 总结用中文，代码片段保持原样
- 不要偷懒跳过代码对比——这是用户最关心的部分
